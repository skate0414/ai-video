# 后端测试验收文档

本文档用于回答一个核心问题：

对于 `ai-video-main/src` 后端代码，什么才算“做成功了”。

这里的“成功”不是只看“代码能跑”或“测试能过”，而是同时满足：

1. 功能正确
2. 接口稳定
3. 流程可恢复
4. 并发安全
5. 质量可控
6. 日志与观测可追踪
7. 回归风险可接受

本文档适用于：

- 日常开发提测
- 联调验收
- PR 自查
- 上线前回归
- 阶段性架构评审

配套执行文档：

- [BACKEND_ACCEPTANCE_WORKFLOW.md](BACKEND_ACCEPTANCE_WORKFLOW.md)

## 1. 成功的定义

一个后端改动要被判定为“成功验收”，至少要同时满足以下 5 类标准：

### 1.1 业务成功

- 用户能通过接口完成预期操作。
- Pipeline 能按预期推进或在应暂停的位置暂停。
- 失败时有明确错误信息，不出现静默失败。

### 1.2 工程成功

- `npx tsc --noEmit` 通过。
- `npx vitest run` 通过。
- 新增或修改逻辑有对应测试覆盖。

### 1.3 运行成功

- 同一项目不会出现并发 run 状态竞争。
- 重启后能从持久化状态恢复。
- 中间产物可追溯，可用于重试与审计。

### 1.4 协作成功

- 路由、服务、编排、阶段、适配器的分层边界未被破坏。
- 文档、测试、接口行为保持一致。
- 新成员能从文档和日志理解这次改动影响。

### 1.5 发布成功

- 不引入高概率回归。
- 不破坏已有主链路。
- 对外行为变化已被记录。

## 2. 验收范围

后端验收覆盖以下模块：

- `server.ts`
- `routes/*`
- `pipeline/*`
- `pipeline/stages/*`
- `pipeline/stages/defs/*`
- `adapters/*`
- `configStore.ts`
- `workbench.ts` 及其相关基础设施（若改动涉及）

## 3. 分层验收标准

## 3.1 路由层验收（`routes/*`）

判定成功标准：

- 请求参数解析正确。
- 错误状态码与错误体语义正确。
- 只调用 `PipelineService` / `Workbench`，不越层访问内部实现。
- 与前端约定的接口结构未漂移。

检查项：

- [ ] 路径、方法、参数名与约定一致
- [ ] 缺参、非法参、空参有明确响应
- [ ] 成功响应字段稳定
- [ ] 异常响应为可读错误，不泄漏敏感信息
- [ ] 没有把业务流程写进 route handler

## 3.2 服务层验收（`pipeline/pipelineService.ts`）

判定成功标准：

- 对路由暴露稳定、可理解的方法。
- 没有向上层泄漏 orchestrator 内部对象。
- 能正确封装配置、项目、provider、导入导出等业务能力。

检查项：

- [ ] 方法语义与命名一致
- [ ] 返回值适合路由直接使用
- [ ] 未暴露底层可变状态对象
- [ ] 对异常进行了必要封装或透传

## 3.3 编排层验收（`pipeline/orchestrator.ts`）

判定成功标准：

- 主流程按 registry 顺序执行。
- 暂停、恢复、重试语义正确。
- 阶段状态、artifact、日志三者一致。
- 关键后置钩子仍然生效（如 QA / 风险闸门 / 关键帧质量控制）。

检查项：

- [ ] `run()` 不依赖硬编码阶段块
- [ ] `retryStage()` 能正确回滚后续阶段状态
- [ ] `resumePipeline()` 不破坏已完成阶段
- [ ] 中断后不会留下不可恢复的脏状态
- [ ] Pipeline 完成、失败、暂停均有明确事件与日志

## 3.4 阶段层验收（`pipeline/stages/*`）

判定成功标准：

- 单个阶段输入输出清晰。
- 依赖通过 `StageRunContext` 注入，不走隐式共享。
- 失败时能定位是输入问题、模型问题还是外部依赖问题。
- 中间产物命名稳定、可读、可恢复。

检查项：

- [ ] 读取的 artifact 存在且字段契约明确
- [ ] 输出 artifact 结构稳定
- [ ] 日志至少包含 stage、动作、结果
- [ ] 不使用无类型逃逸或随意 `as any`
- [ ] 对外部调用失败有超时、重试或兜底策略

## 3.5 适配器层验收（`adapters/*`）

判定成功标准：

- 满足 `AIAdapter` 或对应外部集成契约。
- 第三方失败可识别、可降级、可追踪。
- 不把第三方响应格式污染上传业务层。

检查项：

- [ ] 输入输出已标准化
- [ ] 限流、配额、超时有处理策略
- [ ] 日志中可以分辨 provider 侧错误
- [ ] 对上层暴露统一错误语义

## 4. 功能验收清单

以下是后端“功能成功”的最小验收集：

### 4.1 Pipeline 基础能力

- [ ] 创建项目成功
- [ ] 查询项目成功
- [ ] 删除项目成功
- [ ] 启动 pipeline 成功
- [ ] 停止 pipeline 成功
- [ ] 恢复 pipeline 成功
- [ ] 重试某阶段成功

### 4.2 人工介入能力

- [ ] QA 审核后可人工 override
- [ ] 可修改 script
- [ ] 可修改 scenes
- [ ] 可批准/拒绝 scene
- [ ] 可重新生成单个 scene 资产

### 4.3 配置与能力管理

- [ ] 读取配置成功
- [ ] 更新质量档位成功
- [ ] 更新 TTS 配置成功
- [ ] 更新视频 provider 配置成功
- [ ] 查询 provider capabilities 成功
- [ ] 修改 provider capability 成功

### 4.4 导入导出能力

- [ ] 项目导出成功
- [ ] 项目导入成功
- [ ] 导入后项目可继续运行或查看

## 5. 质量验收清单

后端不仅要“跑通”，还要支撑可控质量。

判定成功标准：

- 低成本阶段先过滤风险，再进入高成本生产。
- 质量闸门有效，不会在明显低质量状态下继续烧资源。
- 用户可以在关键节点介入，而不是只能整条重跑。

检查项：

- [ ] `CAPABILITY_ASSESSMENT` 能拦截高风险主题
- [ ] `QA_REVIEW` 低分会进入人工审查路径
- [ ] `REFERENCE_IMAGE` 后可人工审看视觉方向
- [ ] `KEYFRAME_GEN` 失败有兜底策略
- [ ] `REFINEMENT` 能识别缺失资产或不完整结果
- [ ] 最终风险门仍会执行

## 6. 并发与恢复验收清单

这是后端是否接近生产可用的关键标准。

### 6.1 并发验收

- [ ] 同一 `projectId` 并发启动时，只有一个 run 能成功进入执行
- [ ] 重试与恢复不会绕过锁语义
- [ ] 不同 `projectId` 之间可以并行，不互相污染

### 6.2 恢复验收

- [ ] 服务重启后项目仍可读取
- [ ] 已完成阶段不会重复执行
- [ ] artifact 完整时可 resume
- [ ] 某阶段失败后可局部 retry，而不是整条链路强制重跑

### 6.3 持久化验收

- [ ] `project.json` 与中间 artifact 同步一致
- [ ] 原子写没有留下半截文件
- [ ] 删除项目能清理对应目录

## 7. 可观测性验收清单

判定成功标准：出了问题能定位，而不是“知道失败了但不知道为什么”。

检查项：

- [ ] 关键请求有入口日志
- [ ] 每个 stage 至少有开始/结束/失败日志
- [ ] SSE 能把关键状态实时推到前端
- [ ] 失败日志至少包含 stage、projectId、错误摘要
- [ ] 关键 metrics 可生成并落盘

## 8. 自动化验收标准

必须通过：

```bash
npx tsc --noEmit
npx vitest run
```

判定标准：

- 类型检查 0 error
- 现有测试全部通过
- 新增逻辑必须有新增或更新后的测试

建议补充的自动化项：

- 路由契约测试
- pipelineService 合约测试
- retry/resume/runLock 组合测试
- adapter mock 集成测试

## 9. 手工验收用例模板

每个后端改动提交前，建议至少执行以下手工验收记录：

### 用例 1：主流程成功

- 前置条件：存在一个可运行项目
- 操作：启动 pipeline
- 预期：阶段按顺序推进，关键节点暂停，最终状态正确

### 用例 2：失败后可恢复

- 前置条件：人为制造某阶段失败
- 操作：修复输入或配置后 resume / retry
- 预期：从正确阶段恢复，不重复破坏已完成阶段

### 用例 3：并发保护

- 前置条件：同一项目同时发起两次 start
- 操作：并行请求
- 预期：只有一个执行实例进入运行，另一个明确失败或被拒绝

### 用例 4：人工介入

- 前置条件：流程停在 QA 或 REFERENCE_IMAGE 后
- 操作：修改脚本、场景或执行 override
- 预期：恢复后继续执行且状态一致

```text
- 
```

## 11. 最终验收结论

只有当以下条件同时成立，才可以认为后端改动“成功”：

- 功能正确
- 自动化检查通过
- 手工验收关键路径通过
- 并发与恢复机制未被破坏
- 质量闸门仍然有效
- 文档、日志、测试与实际行为一致

如果只是“本地跑起来了”，但上述任一关键项不满足，那么它不应被定义为验收成功。

如需按仓库现有脚本实际执行，请结合 [BACKEND_ACCEPTANCE_WORKFLOW.md](BACKEND_ACCEPTANCE_WORKFLOW.md) 按顺序完成。
