## 概述

ai-video-main 的核心思路，不是把“输入一个主题，输出一条视频”整个黑盒交给单次模型调用，而是把视频生产拆解成 **13 个主阶段 + 4 个质量子步骤**。每个阶段都有独立的输入、输出、路由策略、日志、持久化产物和失败处理逻辑。

这套设计的目标，不是单纯提升“平均质量”，而是把影响视频质量的关键变量拆开控制：

1. 把事实正确性与创意表达分开处理。
2. 把脚本质量、视觉质量、资产完整性分开验证。
3. 把高成本步骤延后，只在前置质量过关后再投入资源。
4. 把可自动修复的问题交给子步骤处理，把高风险问题拦截在合成前。
5. 把每个中间结果保存为产物，使得用户可以审查、编辑、恢复、重试。

因此，ai-video-main 的“可控质量”并不是来自某个更强的单一模型，而是来自一套 **分阶段约束 + 子步骤校验 + 人在回路 + 持久化恢复** 的生产系统。

## 为什么这套流水线能实现可控质量

传统一键式视频生成有四个典型问题：

1. 错了不知道错在哪一层：是事实错、脚本弱、分镜乱，还是素材没对齐。
2. 错了不能局部修：通常只能改 prompt 重来。
3. 成本前置：在脚本、分镜还不稳定时就开始烧图片和视频额度。
4. 一致性难以保证：脚本、视觉、音频常常各自漂移。

ai-video-main 通过以下方式解决：

1. **先约束，再生成**：先提取 Style DNA、做事实研究、做叙事规划，再生成脚本和画面。
2. **先低成本过滤，再高成本生产**：先做文本和结构审查，再做图片，再做视频。
3. **先建立锚点，再批量扩散**：先生成参考图与 Reference Sheet，再去做关键帧和视频。
4. **每一层都有护栏**：安全门、CV 预处理、脚本自审、主体隔离、终审风控门依次收紧误差范围。
5. **允许人工介入**：在 QA_REVIEW、STORYBOARD、REFERENCE_IMAGE 后自动暂停，用户可以审稿、改分镜、选图、再恢复。

## 总体控制框架

从实现上，整条流水线由 `PipelineOrchestrator` 编排。它负责：

1. 按顺序执行 13 个主阶段。
2. 在关键阶段后自动暂停。
3. 为每个阶段选择合适的适配器和模型。
4. 将每个阶段的产物保存到 `data/projects/{projectId}/`。
5. 在失败或服务重启后，从已完成阶段继续执行。

这意味着，质量控制不是散落在各个 prompt 里的软约束，而是被编排器上升为系统行为。

### 控制链路

```text
主题输入
  -> 安全与能力检查
  -> 风格真值提取
  -> 事实研究
  -> 叙事与字数规划
  -> 脚本生成
  -> 脚本自审与 QA
  -> 分镜生成
  -> 视觉可拍摄性检查
  -> 参考图与风格锚定
  -> 关键帧
  -> 视频片段
  -> 语音
  -> FFmpeg 合成
  -> 终审风控门
  -> 自动精修
```

这条链路的关键在于：后面的高成本阶段，不会在前面的低成本控制失效时盲目继续。

## 13 个主阶段如何逐步收敛质量

### 1. CAPABILITY_ASSESSMENT：先做安全与可行性筛查

这是第一道硬门。系统先判断主题是否存在高风险内容，例如医疗误导、自残、自杀、仇恨等。如果安全检查失败，直接抛出 `SafetyBlockError`，流水线终止。

它带来的“可控”是：

1. 避免系统在不安全主题上继续投入任何资源。
2. 把安全问题前置，而不是等到视频已经生成出来再补救。
3. 将“能不能做”与“怎么做好”分离。

### 2. STYLE_EXTRACTION：先把风格约束结构化

该阶段把参考视频或主题抽取为 `StyleProfile`，包括：

1. 视觉风格、配色、光影、节奏。
2. 目标受众、叙事结构、hook 与 CTA 类型。
3. 三轨 Style DNA：脚本轨、视觉轨、音频轨。

它的作用不是生成内容，而是把后续所有生成共享的“风格约束”固定下来。后面脚本、分镜、TTS、关键帧都以它为条件，而不是各自自由发挥。

### 3. RESEARCH：把事实正确性交给独立阶段处理

脚本是否可信，不能只依赖创作模型的常识。RESEARCH 阶段独立收集事实、来源与可信度，为后续脚本提供外部依据。

这样做的价值在于：

1. 让脚本生成阶段以“事实材料”为输入，而不是临场编造。
2. 把事实研究与文案表达解耦，降低幻觉传递。
3. 给 QA 审查和后续人工检查留下可追溯依据。

### 4. NARRATIVE_MAP：先规划叙事，再写正文

NARRATIVE_MAP 包含两个层面：

1. `Calibration`：根据目标时长、语速、结构密度，计算目标字数、区间、段落分配。
2. `Narrative Map`：先把脚本骨架拆成开场、推进、转折、结尾等结构。

它控制的是“篇幅”和“叙事秩序”。

如果没有这一层，脚本生成容易出现两类问题：

1. 太长或太短，最终时长失控。
2. 结构松散，后续分镜数量与视频节奏失衡。

### 5. SCRIPT_GENERATION：在受约束条件下写脚本

SCRIPT_GENERATION 不是裸写作，而是在以下约束下生成：

1. 风格 DNA。
2. 研究事实。
3. 校准数据与字数预算。
4. 叙事地图。
5. 生成计划。

这一步的可控点在于：脚本从一开始就被限制在“风格正确 + 结构正确 + 时长合理 + 内容有依据”的框架里。

### 6. QA_REVIEW：在进入视觉生产前做集中审查

QA_REVIEW 是整条流水线的第一个大质量闸门。它对脚本执行 **4 维度审计（每维度 1-10 分）**，而不是等到视频成片后再发现问题。

4 个审计维度：

1. **Accuracy & Factual Integrity（事实准确性）**：检查编造数据、误导性简化、危险的医疗/健康断言、未标注来源的数值声明。可疑数值会被提取到 `suspiciousNumericClaims` 列表中。
2. **Style Consistency（风格一致性）**：对照 Style DNA 检查语气、句长、hook 策略、情感弧线、隐喻数量、互动线索、术语一致性。偏差记录在 `styleDeviations` 中。
3. **Production-Readiness（可拍摄性）**：检查每个句子能否被独立渲染为画面、节奏是否合适、CTA 是否自然、字数是否在目标区间。不可拍摄句子列入 `unfilmableSentences`。
4. **Content Contamination（内容污染）**：将生成脚本与参考视频原始转录做比对，检测逐字抄袭（>8 字符）、复用事实/数据点、复用视觉隐喻、原主题术语泄露。满分 10 意味着脚本与参考仅共享风格而非内容。

系统根据 `overall_score` 自动判定：**>= 7 分自动通过**，< 7 分标记需人工审查。

该阶段后自动暂停（默认 `pauseAfterStages` 之一），用户可以：

1. 查看 QA 结果与各维度分数。
2. 直接编辑脚本。
3. 手动覆盖 QA 决策。
4. 再恢复流水线。

这使“质量控制”从纯自动化，升级为 **模型检查 + 人工裁决** 的双层机制。

### 7. STORYBOARD：把脚本变成可执行的视觉计划

分镜阶段把文本脚本转为 `Scene[]`，每个场景带有：

1. `narrative`
2. `visualPrompt`
3. `estimatedDuration`
4. `productionSpecs`
5. `assetType`
6. `id`、`number`、`status`、`logs`

这一步将抽象文案变成视觉生产可消费的结构化任务单。质量不再只停留在"脚本听起来不错"，而是要进入"每个句子能否被正确拍出来/生成出来"的层面。

该阶段完成后，系统自动暂停（默认 `pauseAfterStages` 之一），用户可以在此检查分镜结构、编辑场景、调整 `visualPrompt` 或 `assetType`，确认后再恢复流水线进入视觉生产。

### 8. REFERENCE_IMAGE：先看参考图，再决定是否进入批量生成

REFERENCE_IMAGE 的角色不是直接产出最终素材，而是产出“供审查的视觉预演结果”。

现在该阶段还引入了 **Reference Sheet**：

1. 先为整个项目生成一张风格参考总图。
2. 把这张图作为 `referenceImage` 传给 Gemini 图像生成。
3. 让每个场景图像都基于同一个视觉锚点展开。

该阶段完成后同样自动暂停（默认 `pauseAfterStages` 之一），用户可以逐场景审查参考图、重新生成不满意的场景、或调整视觉方向后再恢复。

这样做的可控点在于：

1. 先检查画风是否跑偏。
2. 先检查角色、配色、材质、光影是否稳定。
3. 在还没进入视频生成之前，以低成本修正视觉方向。

### 9. KEYFRAME_GEN：在风格锁定后生成可用于视频的关键帧

关键帧阶段只为 `assetType === 'video'` 的场景生成图像，并且会优先复用参考图作为兜底。该阶段有明确的重试策略：最多 2 次重试（共 3 次尝试），退避间隔为 `5000ms × 重试次数`。

具体的降级链路如下：

1. 若 `keyframeUrl` 已存在 → 跳过。
2. 若 `referenceImageUrl` 存在 → 直接用参考图作为关键帧（免费兜底）。
3. 调用 `adapter.generateImage()` 生成新关键帧 → 成功则写入 `keyframeUrl`。
4. 若所有尝试失败 → 最终回退到 `referenceImageUrl`。

系统会分别统计 `freshKeyframes`（新生成）和 `fallbackKeyframes`（参考图兜底），便于质量诊断。

这一步的可控性体现在：

1. 视频不是凭空生成，而是基于关键帧进入 img2video。
2. 如果关键帧生成失败，仍有参考图兜底，避免整段流程中断。
3. Reference Sheet 会继续参与关键帧生成，减少跨场景漂移。

### 10. VIDEO_GEN：把运动预算投到真正值得动的场景上

VIDEO_GEN 只处理被标记为视频的场景，不要求所有场景都变成视频。这是一个重要的质量与成本平衡设计。

系统通过：

1. `assetType` 区分 image / video。
2. 最低视频场景数约束（`MIN_VIDEO_SCENES`，默认 2），保证成片有足够动效。如果不足，STORYBOARD 阶段会自动将最长的非视频场景提升为视频类型。
3. 多账号并发策略：单账号时顺序处理避免锁冲突；多账号时使用 worker pool + round-robin 分配，每个 worker 绑定独立 profile，重试时自动轮转到下一个 profile（`(workerIdx + attempt) % profileCount`）。
4. 风格锚点 `styleAnchor`，把前一个成功场景的风格摘要传给后续视频生成。

这样做避免了“所有镜头都硬转视频，导致质量下降和成本爆炸”的问题。

### 11. TTS：把配音质量从视频质量中独立出来控制

TTS 阶段为每个场景旁白独立合成语音，记录音频路径和时长。并且会根据风格自动解析音色与语速。

它的作用是：

1. 让声音节奏受风格与校准约束。
2. 让后续合成按真实音频长度对齐。
3. 避免“画面做完了，配音节奏又完全不是一回事”的割裂。

### 12. ASSEMBLY：把素材正确拼成成片，而不是简单拼接

ASSEMBLY 用 FFmpeg 对所有场景素材进行标准化处理，再完成：

1. 图片/视频归一化。
2. 按顺序拼接。
3. 旁白叠加。
4. 时长对齐。
5. 背景音乐混合。

这是最终“成片质量”真正落地的一步。前面每层控制的是内容质量，这一步控制的是交付质量。

### 13. REFINEMENT：不要把失败场景静默吞掉

REFINEMENT 阶段负责检查失败场景、自动重试、必要时重新合成，并在完成前执行终审风控门。

它的价值在于：

1. 将单场景失败隔离，而不是整片报废。
2. 将失败显式暴露在结果中，而不是静默产出残缺视频。
3. 把“是否可以宣告完成”交给明确规则，而不是凭流程走完就算成功。

## 4 个质量子步骤如何作为护栏

13 个主阶段定义了生产主线，4 个质量子步骤则像“护栏”，专门用来识别和修正容易在主线中被忽略的问题。

### 1. CV 预处理：用真实视觉特征纠偏

CV 预处理位于 STYLE_EXTRACTION 之前或之中，它会：

1. 用 FFmpeg 抽取缩略图。
2. 分析主色、亮度、人脸比例等低层视觉特征。
3. 用这些结果覆盖 LLM 纯推断得到的配色。

这是从“模型猜参考视频长什么样”升级到“系统先看一眼真实视频，再给模型约束”。

### 2. 脚本自审：用第二次模型调用审第一稿

SCRIPT_GENERATION 完成后，系统不会立刻接受结果，而是启动 `Script Audit`：

1. 检查字数区间是否达标（是否落在 `target_word_count_min` ~ `target_word_count_max` 之间）。
2. 检查事实是否有风险（未标注来源的数值声明、编造统计）。
3. 检查语气、句长、隐喻密度是否偏离 Style DNA。
4. 检查每句是否可独立视觉化（能否被渲染为画面）。
5. 检查安全合规（绝对化的医疗/健康断言、编造数据）。
6. 输出 `styleConsistencyScore`（0-1）与修正稿 `correctedScript`。

判定阈值：**`styleConsistencyScore >= 0.78` 为 pass**，低于 0.78 标记为 warn，提醒存在风格偏移。

这一步相当于让模型扮演“第二位编辑”，降低首稿质量波动。

### 3. 主体隔离验证：确保每一帧都可生成

分镜常见的问题不是“句子不好看”，而是“画面生成模型不知道主角是谁”。主体隔离验证专门检查：

1. 有没有清晰主主体。
2. 描述是否足够具体。
3. 是否有多个竞争主体导致画面发散。
4. 抽象概念是否被转化为可视隐喻。

如果失败，系统会直接回写更清晰的 `revisedPrompt`。

这一步把“视觉可拍摄性”从经验判断，变成可执行的自动修正。

### 4. 终审风控门：在交付前做最后硬校验

最终风险门在 REFINEMENT 之前运行，执行 3 项硬校验：

1. **场景完整性（sceneCompleteness）**：检查每个场景是否都有视觉资产（`assetUrl` 存在且 `assetType !== 'placeholder'`）和音频资产（`audioUrl` 存在）。
2. **占位文本检测（placeholderDetection）**：用正则扫描所有场景的 `narrative` 和 `visualPrompt`，检测 `[TODO]`、`[INSERT]`、`[PLACEHOLDER]`、`PLACEHOLDER`、`lorem ipsum`、`TBD` 等占位内容。
3. **叙事安全检查（narrativeSafety）**：对最终合成脚本重新运行 `runSafetyMiddleware`，检测自杀风险、医疗声明等安全问题。

前两项为非阻塞检查（结果保存到 `final-risk-gate.json`，警告记录到日志），但第三项——安全问题——是硬阻塞：如果检测到安全违规，直接抛出 `SafetyBlockError`，流水线终止。

## “可控”不是口号，而是工程机制

ai-video-main 之所以能够说自己是“可控质量”，关键不是流程图好看，而是以下机制真实存在于实现中。

### 1. 阶段级持久化

每个阶段都会把结果保存为 JSON 产物，例如：

1. `style-profile.json`
2. `research.json`
3. `calibration.json`
4. `narrative-map.json`
5. `script.json`
6. `qa-review.json`
7. `scenes.json`

这样用户可以明确知道哪一步产出了什么，也能对单个产物做诊断和覆盖。

### 2. 阶段级状态机

每个阶段都有 `pending / processing / completed / error` 状态。编排器据此决定：

1. 哪一步可以跳过。
2. 哪一步需要重试。
3. 恢复时该从哪一步继续。

这就是“可恢复”的基础，也是“质量问题可局部定位”的前提。

### 3. 自动暂停与人在回路

项目默认在以下阶段后暂停：

1. `QA_REVIEW`
2. `STORYBOARD`
3. `REFERENCE_IMAGE`

暂停的价值是把用户审查放在高成本步骤之前。与其让用户在最终视频里发现问题，不如让用户在脚本和参考图阶段就截断问题。

### 4. 质量路由与成本路由耦合

系统并不是“所有任务都用同一个最强模型”，而是按任务类型和资源稀缺度路由：

1. 文本分析类任务优先走免费聊天。
2. 图片类任务免费优先，额度耗尽再自动降级到 API。
3. 视频类任务优先分配更稳定的付费能力。

这样做本质上也是质量控制的一部分，因为高成本步骤用更稳定资源，能减少关键路径上的失败率。

### 5. 会话上下文复用

系统把 13 个阶段拆到 4 组会话中，组内共享上下文：

1. **Analysis 组**：CAPABILITY_ASSESSMENT → STYLE_EXTRACTION → RESEARCH，共享分析上下文。
2. **Creation 组**：NARRATIVE_MAP → SCRIPT_GENERATION → QA_REVIEW，保持叙事连贯性。
3. **Visual 组**：STORYBOARD → REFERENCE_IMAGE → KEYFRAME_GEN，维持视觉风格一致。
4. **Production 组**：VIDEO_GEN、TTS、ASSEMBLY、REFINEMENT，各阶段独立会话。

这样做能减少“同一个项目里前后说法不一致”的漂移问题。

### 6. 视觉锚定机制

最近引入的 Reference Sheet 机制，进一步强化了视觉一致性：

1. 先生成全局风格总图。
2. 在参考图和关键帧生成时，把这张图作为 Gemini 的多模态输入。
3. 让每个画面基于同一个“看得见的风格锚点”生成，而不只是靠文本反复描述。

这说明 ai-video-main 的可控性不是静态的，而是在持续增强关键质量闭环。

## 各阶段产物如何保证可追溯与可恢复

这套系统把每一阶段都视为一个可审计节点，而不是瞬时过程。每个节点至少具备以下属性：

1. 明确输入。
2. 明确输出。
3. 可单独记录日志。
4. 可持久化。
5. 可单独重试。
6. 可作为后续阶段的唯一事实来源。

这带来三个结果：

1. **可追溯**：最终视频出问题时，能回溯到脚本、分镜、参考图还是关键帧。
2. **可恢复**：服务重启或中断后，不必整条链重新跑。
3. **可干预**：用户能在中间插入人工编辑，并让后续阶段继续消费修改后的产物。

## 对比一键式视频生成方案的本质差异

一键式方案的核心优势是快，但缺点是：

1. 质量问题难拆解。
2. 修改成本高。
3. 风格稳定性弱。
4. 事实可靠性难保障。
5. 最终交付经常是“能看”，而不是“可生产”。

ai-video-main 走的是另外一条路：

1. 先把问题拆小。
2. 再对每一层定义输入、输出与质量门。
3. 在低成本阶段尽可能发现问题。
4. 在高成本阶段只处理已经通过前置检查的内容。
5. 在最后再用风险门决定是否允许交付。

因此，它更接近一套 **视频生产工程系统**，而不是一个“调用几个模型凑在一起”的生成器。## 观测性与质量度量

除了阶段级产物和状态机之外，系统还内置了一套观测性（Observability）框架，让每次流水线运行的质量不仅"可感知"，而且"可量化、可对比"。

### 阶段级度量（StageMetrics）

每个阶段会自动记录：

1. `startedAt` / `completedAt`：精确的开始和结束时间戳。
2. `durationMs`：该阶段耗时（毫秒）。
3. `status`：`running` / `completed` / `error`。
4. `llmCallCount`：该阶段消耗的 LLM 调用次数。
5. `estimatedTokens`：估算 token 消耗。
6. `qualityScore`：该阶段的质量分数（0-1），由审计步骤填写。

### 流水线级度量（PipelineMetrics）

汇总所有阶段的度量，提供：

1. `totalDurationMs`：整条流水线总耗时。
2. `totalLlmCalls`：所有阶段 LLM 调用总次数。
3. `overallQualityScore`：所有阶段质量分数的均值。
4. `stages`：按阶段名索引的完整度量记录。

这套度量的价值在于：

1. **成本诊断**：哪些阶段消耗了最多 LLM 调用？哪些阶段最慢？
2. **质量趋势**：多次运行后可以比较不同项目、不同主题的质量分布。
3. **故障定位**：阶段级错误状态和耗时异常可以快速定位瓶颈。

## 基础设施层：原子持久化与并发安全

"可控质量"不仅需要流程设计正确，还需要底层基础设施可靠。以下两个机制保障了数据完整性和并发安全：

### 原子写入（ProjectStore）

所有项目数据的持久化都通过 `ProjectStore` 完成，采用 **write-to-temp-then-rename** 模式：

1. 先将数据写入临时文件（随机文件名，避免冲突）。
2. 再用 `renameSync` 原子替换目标文件。

这样做的好处是：即使写入过程中进程崩溃或断电，也不会出现"文件写了一半"的损坏状态——要么是旧文件完整，要么是新文件完整。

### 并发锁（RunLock）

`RunLock` 是一个 per-project 的并发守卫，防止同一个项目被同时运行两次：

1. `run()` 方法在启动前调用 `runLock.acquire(projectId)`，获取锁。
2. 如果该项目已有运行中的流水线，`acquire` 返回 `false`，`run()` 抛出"项目已在运行"错误。
3. 流水线正常完成或异常终止时，`runLock.release(projectId)` 释放锁。
4. 用户可以通过 `runLock.abort(projectId)` 中止特定项目的运行。

这解决了一个常见的生产环境问题：用户误触"再次运行"或 API 重复调用导致同一项目的两条流水线竞争写入同一组文件，最终产出损坏数据。

## 结论

ai-video-main 之所以能实现"可控质量的视频生成"，本质上依赖的是六件事：

1. **将问题分层**：把风格、事实、叙事、分镜、素材、交付拆成独立阶段。
2. **将风险前移**：用安全门、研究、QA、主体隔离、参考图审查，在高成本步骤之前发现问题。
3. **将质量显式化**：通过分数（QA overall_score >= 7、脚本自审 styleConsistencyScore >= 0.78）、检查结果、日志、产物和状态，把质量从主观感觉变成可检查对象。
4. **将人工纳入闭环**：在 QA_REVIEW、STORYBOARD、REFERENCE_IMAGE 三个关键节点暂停，允许人工覆盖模型判断。
5. **将流程产品化**：所有中间结果可持久化、可恢复、可单独重试。
6. **将基础设施可靠化**：通过原子写入保证数据完整性，通过并发锁保证运行安全，通过观测性度量保证质量可追踪。

这就是 ai-video-main 与普通一键式 AI 视频工具最根本的不同。它不是把“生成视频”当作一次模型调用，而是把它当作一条需要工程化治理的生产线。
